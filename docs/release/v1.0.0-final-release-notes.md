## Visão Geral

O **SRM Credit Engine** é uma plataforma de backoffice financeiro para operações de cessão de crédito em FIDC (Fundo de Investimento em Direitos Creditórios). A v1.0.0 representa a **entrega completa do desafio técnico**, consolidando 13 milestones incrementais que cobrem domínio financeiro, API REST, frontend Angular, observabilidade, documentação de system design, gestão de crise, pipeline CI/CD e fechamento da release.

O sistema implementa o fluxo completo de operação de mesa financeira:

```
Recebível → Precificação → Câmbio → Liquidação Transacional → Auditoria
```

## Arquitetura

**Padrão:** Monólito modular em 5 camadas com fronteiras explícitas

```
interfaces/rest   → Controllers, DTOs, tratamento de exceções HTTP
application       → Casos de uso, commands, orquestração de transações (@Transactional)
domain            → Entidades, strategies de precificação, serviços de domínio puro
infrastructure    → JPA, JdbcTemplate, configurações, observabilidade
reporting         → Queries analíticas com SQL nativo (NamedParameterJdbcTemplate)
```

**Decisões arquiteturais registradas:** ADRs 0001 a 0012 em `docs/adr/`

## Backend

### Funcionalidades Implementadas

| Funcionalidade | Detalhe |
|---|---|
| Schema financeiro | 7 tabelas com constraints NUMERIC, UNIQUE, CHECK e índices |
| Motor de precificação | Strategy Pattern — 2 estratégias: Duplicata (0,4% a.m.) e Cheque Pré-Datado (1,0% a.m.) |
| Motor de câmbio | Pares direcionais BRL→USD e USD→BRL, taxa positiva obrigatória |
| Liquidação transacional | ACID com 3 barreiras: status check + UNIQUE(receivable_id) + @Version |
| Snapshot cambial | Taxa do momento da liquidação gravada imutavelmente em `settlements` |
| Outbox Pattern | Evento persistido na mesma transação — publisher não implementado |
| API REST | 5 endpoints com validação Bean Validation e tratamento de exceções HTTP |
| Relatórios | SQL nativo com filtros dinâmicos, paginação cursor-based e ordenação |
| Observabilidade | 5 contadores + 3 timers via Micrometer + Prometheus |
| OpenAPI/Swagger | UI interativa em `/swagger-ui/index.html` |

### Precisão Financeira

- `BigDecimal` em todos os valores monetários, taxas e câmbio — sem `double` ou `float`
- `RoundingMode.HALF_EVEN` para arredondamento bancário (IEEE 754)
- `NUMERIC(19,4)` para valores financeiros; `NUMERIC(19,10)` para taxas de câmbio

### Testes

- **105 testes** unitários e de integração (Spring Boot Test + Mockito)
- Cobertura JaCoCo: **96,9% de linhas** — acima do threshold de 90%
- Pre-push hook local: `scripts/pre-push.sh` bloqueia push com build/testes/cobertura quebrados

## AI_USAGE — Modelo de Desenvolvimento

O projeto utilizou um modelo estruturado de **desenvolvimento assistido por IA com subagentes especializados**:

- 7 subagentes definidos em `agents/` (Analista, Arquiteto, Backend, Frontend, DevOps, QA, Orquestrador)
- 14 milestones (v0.2.0 → v1.0.0), cada uma com: prompt → plano → aprovação humana → implementação → validação
- Nenhuma decisão técnica tomada de forma autônoma — toda implementação precedida de plano aprovado pelo desenvolvedor

Análise completa, decisões aceitas, rejeitadas e lições aprendidas: `AI_USAGE.md`

## Limitações Conhecidas

| Limitação | Observação |
|---|---|
| Sem autenticação | Acesso aberto para fins do desafio — JWT ou OAuth2/OIDC seria o próximo passo |
| Apenas BRL e USD | Seed com 2 moedas; framework suporta extensão sem mudança de schema |
| Apenas 2 tipos de recebível | DUPLICATA e CHEQUE_PRE_DATADO; Strategy Pattern facilita adição |
| Taxas de câmbio manuais | Sem integração com API externa (ex: Open Exchange Rates) |
| Dispatcher do Outbox não implementado | Eventos persistidos em `outbox_events`; publicação (Kafka/RabbitMQ) não implementada |
| Sem Grafana | Prometheus configurado e scraping; dashboard de visualização não incluído |
| Sem Resilience4j | Planejado; nenhuma dependência externa real exige resiliência neste escopo |
| Sem Testcontainers | Testes usam mocks e Spring Boot Test; banco real não coberto por testes de integração |
| Push de imagem Docker não implementado | CI faz build das imagens; push para registry é evolução futura |

## Próximos Passos Naturais

1. Dispatcher do Outbox — worker com polling ou CDC (Debezium)
2. Autenticação — JWT ou OAuth2/OIDC com Spring Security
3. Publicação de imagens Docker em registry (CI de build já existe)
4. Grafana — dashboard de negócio com as 8 métricas existentes
5. Testcontainers — testes de integração com PostgreSQL real
6. Integração com API externa de câmbio
7. Rate limiting e proteção contra abuso
8. Evolução para CQRS com read replica para relatórios

## Comandos para Tag v1.0.0

A tag deve ser criada **manualmente** pelo desenvolvedor, após merge da branch de release na `main`:

```bash
# Após merge do PR na main
git checkout main
git pull origin main

# Criar e publicar a tag
git tag -a v1.0.0 -m "SRM Credit Engine v1.0.0 — Final release: 13 milestones, full FIDC receivable credit platform"
git push origin v1.0.0
```

**Importante:** a tag nunca deve ser criada antes do merge na `main`.
